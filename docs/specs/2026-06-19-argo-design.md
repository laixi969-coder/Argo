# 金羊毛 Argo — 私人选品雷达 · 设计文档

> 名字溯源：希腊神话 Argo 之船远征寻找金羊毛。中文「金羊毛」= 宝物（赚钱产品），英文「Argo」= 寻宝的船（扫描雷达）。

- 日期：2026-06-19
- 状态：已与蔡蔡确认，待进实现计划

## 1. 目标与边界

每天自动扫公开数据源，找「真需求 + 有人愿掏钱」的产品机会（含实体 / AI / 虚拟 / 服务），
用 `/req` 真需求框架过滤掉伪需求，每天早上发一封 Top 20 日报邮件到蔡蔡的 Gmail。

**服务对象**：蔡蔡本人，私人选品 / 投资决策雷达。只要榜单准，不做对外产品。

**明确不做**（YAGNI）：对外网页、多用户账号体系、数据库、订阅、规模化。核心交付是一封私人日报；后续按蔡蔡要求增加一个仅限本机访问的超级管理员设置页，用于 API 配置、模型同步与连接测试。

### 本机超级管理员设置页

- 地址：`http://127.0.0.1:8765`，服务不得监听公网地址。
- 超级管理员密码仅保存 PBKDF2 哈希；登录会话使用 HttpOnly、SameSite Cookie 与 CSRF 令牌。
- 支持 LLM Base URL、API Key、模型配置，并从兼容接口的 `/models` 同步模型。
- 支持 LLM、Reddit、Product Hunt、SMTP 独立连接测试；SMTP 测试只登录，不发送邮件。
- TikHub 使用官方 `https://api.tikhub.io`，以 `Authorization: Bearer` 鉴权；连接测试调用 `/api/v1/tikhub/user/get_user_info`，同时检查 Key 状态、账户状态、余额和免费额度，不调用社媒数据接口。
- 密钥永不回显；秘密字段留空保存时保留原值。

## 2. 首版范围（已确认 A 方案：先 2 源跑通）

首版只接 **2 个数据源**，把「扫 → 抽取 → 粗筛 → 真需求精判 → 排序 → 发邮件」整条链路跑通、
把打分校准对，之后再逐个加源（Kickstarter 等实体维度、Google Trends、TikTok、广告库）。

### 数据源（覆盖全品类，含 AI / 虚拟 / 服务）

| 源 | 反映什么 | 获取方式 |
|----|---------|---------|
| Reddit（r/SomebodyMakeThis、r/Entrepreneur 等求购/愿付费帖） | 真需求 + 未被满足 | Reddit 官方免费 API（需 key） |
| Product Hunt（每日新品 + upvote/评论热度） | 正在涌现 + 市场反应 | Product Hunt 官方免费 API（需 key） |

竞争密度维度首版**不做**（Reddit/PH 给不了，硬塞会污染榜单），等接入 Amazon 类源再补。

## 3. 数据流

```
[扫源] Reddit + Product Hunt
   ↓ 抓原始帖子/产品（标题、正文、热度、链接）
[证据抽取] 提炼产品机会 + 客户/任务/历史行为/替代方案/已付成本/WTP 证据
   ↓
[粗筛] 按源头信号（求购热度 / upvote）排序 → 砍到 Top 30
   ↓
[真需求精判] 只对 30 个跑 /req 蒸馏版打分
   ↓ 输出：真需求 / 待验证 / 伪需求 + 理由 + 综合分
[过滤+排序] 踢掉伪需求，剩下按分排序取 Top 20
   ↓
[发邮件] 一封 HTML 日报到 Gmail
```

每环节为独立小模块，单独可测可换（见 CLAUDE.md 目录约定）。

## 4. 真需求精判（/req 框架蒸馏版）— 已确认 A 方案：两段式

避免对几百个垃圾候选做深度盘而烧爆 token：

1. **粗筛（便宜）**：源头信号排序，砍到 Top 30。
2. **真需求精判（LLM）**：只对 Top 30 跑一个把 `/req` 四道闸门压成结构化评分的 prompt，每个候选输出：
   - 判定：`真需求 / 待验证 / 伪需求`
   - 价值闸门、共识闸门、模式闸门各一项评分
   - 一句话理由（必须引用原始证据，命中哪面镜子 / 哪条公理）
   - 证据强度，以及一个验证真实支付意愿的最小动作
   - **伪需求直接踢出榜单**，只留真需求 + 有潜力的待验证。

### 证据纪律

- 妈妈测试优先看过去真实行为，不把赞美、未来意向或产品 upvote 当作付费证据。
- 抽取时保留用户当前替代方案、已付出的时间/金钱、发生频率，以及预算/预付/付款记录。
- 原文没有的信息统一标为「未知」；评分模型不得补全或猜测。
- 证据不足的合理点子判为「待验证」，日报同时给出下一步最小验证动作。

`/req` 框架来源：`~/.claude/skills/req/SKILL.md`（价值→共识→模式→求真四道闸门 + 八条公理 + 三面镜子）。

## 5. 关键技术决策

- **运行环境**：蔡蔡 Mac 上的定时任务（launchd / cron），每天早上 7 点跑。无服务器、零托管成本。
  - 风险：Mac 关机则当天不跑。若实测常关机，改用云函数（后续决定，不在首版）。
- **Reddit / Product Hunt**：各申请一个官方免费 API key（蔡蔡操作，提供逐步指令）。
- **真需求精判模型**：默认使用 DeepSeek 官方兼容接口（`https://api.deepseek.com` + `deepseek-chat`），可在本机超级管理员设置页同步并切换模型。
- **语言**：Python 单脚本项目。
- **密钥**：全部 `.env`，绝不进代码 / commit / 日志。

## 6. 错误处理与降级

- 某源挂掉（限流 / 改版）：跳过该源，用另一个源照常出日报，邮件标注「今天 X 没抓到」，不崩流水线。
- 真需求精判失败：降级用源头信号排序，邮件标注「今日未经真需求精判」。
- 无任何数据：发一封「今日无数据」邮件，不静默失败。

## 7. 验证

- `demo` 自检：喂一批假数据，断言打分 / 过滤 / 排序 / 邮件渲染正确（assert 式，无框架）。
- 单元测试：粗筛排序、伪需求过滤、降级路径各一个最小用例。
- 首次真实运行：当场给蔡蔡看一封样例邮件，由其确认榜单质量后再上定时任务。
